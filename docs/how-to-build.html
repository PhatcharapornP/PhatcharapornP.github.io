How To Build This Portfolio

You. Make website. No framework. Just three files and some pictures. This is the whole path from empty folder to a live site that looks like a video-game HUD, in order, with nothing assumed. Never seen code in your life? The ★ section teaches you to read HTML and CSS in five minutes — it sits right before you type anything.

A

Get tools

  • JetBrains Rider (or any editor) — you write text files in it. That's all a website is.
  • GitHub Desktop + a github.com account — this is how the site gets on the internet, for free.
  • A browser — this is how you look at what you made.
No Node, no React, no build tools. A website is text files. The browser reads them directly. Everything else is optional complexity we are not buying.
B

Make the magic repo

  • On github.com: create a new repository named exactly YOURUSERNAME.github.io.
  • That exact name is the trick: GitHub automatically serves whatever is in it at https://YOURUSERNAME.github.io/. This is called GitHub Pages. No server, no hosting bill.
  • In GitHub Desktop: Clone repository → pick it → now there's a folder on your computer connected to the internet one.
  • Open that folder in Rider: File → Open → select the folder. Rider treats a plain folder as a project just fine.
★

Before you type: read the two languages

The next steps ask you to type code. First learn to read it. A website is only two languages, and neither is scary. Five minutes here and every code block below stops looking like magic runes.

The whole idea in one breath

HTML builds bare boxes and puts your words in them. CSS is a list of paint rules for those boxes. One line links them. That is the entire craft — everything else is repetition.

HTML = the boxes (structure)

A web page is boxes stacked and nested inside boxes. HTML draws them with tags. A tag looks like this:

<h1>My Name</h1>
  • <h1> — the opening tag: "a big heading starts here."
  • My Name — the content a human actually sees.
  • </h1> — the closing tag. The slash / means "the heading ends here."

Four rules cover almost all of it:

  • Most tags come in a pair (open + close). Whatever sits between them is what shows up.
  • A few tags are loners — they hold nothing and never close: <img> (a picture), <link>, <meta>.
  • Boxes go inside boxes: <div><p>hi</p></div> is a box holding a paragraph.
  • Tags can carry extra notes called attributes, written inside the <>:
<a href="about.html">About</a>      <!-- href = WHERE the link goes -->
<img src="cat.jpg" alt="a cat">     <!-- src = WHICH picture, alt = describe it -->
<div class="card"> ... </div>        <!-- class = a NICKNAME (the important one) -->
class="card" is a nickname you stick on a box so the paint file can find it later. This one attribute is how HTML and CSS hold hands — remember it above all else.

Every page has the same skeleton: a head (invisible setup) and a body (everything you see):

<head>   ← page title, font links, the link to your style file. Nobody SEES this.
<body>   ← headings, text, pictures, cards. Everything a human looks at.

The dozen tags you will actually use:

  • <h1> <h2> <h3> — headings, biggest to smaller.
  • <p> — a paragraph of text.
  • <div> — a plain box. The workhorse; it holds other things.
  • <span> — a small inline piece (to colour one word inside a sentence).
  • <a> — a link · <img> — a picture.
  • <ul> + <li> — a bullet list and each bullet in it.
  • <button> — a clickable button.

CSS = the paint (looks)

HTML gave you bare boxes. CSS is a list of rules, each one saying "find these boxes, make them look like this." One rule:

.card {
  background: #14171D;
  padding: 1rem;
}
  • .card — the selector: WHO to paint.
  • { } — the paint goes inside the curly braces.
  • background: #14171D; — one instruction: property : value ;. Every line ends in a semicolon.

How to point at boxes (selectors):

  • h1 { } — every h1 heading on the page.
  • .card { } — every box wearing class="card". The dot means "nickname."
  • body { } — the whole page.

The paint instructions (properties) you will use most:

  • color — text colour · background — fill behind the box.
  • padding — cushion inside · margin — gap outside.
  • border — the outline line · font-family — which font.
  • width / max-width — how wide · display: flex — lay the children in a row.

Knowing the box model is most of not fighting CSS. Every box is four nested layers, from the inside out:

MARGIN — gap OUTSIDE (space to neighbours)
BORDER — the visible outline
PADDING — cushion INSIDE
CONTENT
your text / picture

Colours like #E8752E are just a colour's secret code (called "hex"). You never memorise them — you copy them.

Variables are the one clever trick, and they save your life. Name a colour once at the very top:

:root {
  --accent: #E8752E;   /* name the orange once */
}

Then anywhere you want it, write var(--accent) instead of the code. Change the colour in that one place and it updates on every page at once. That is why the whole site shares a single look.

How the two hold hands

HTML:  <div class="card">...</div>                 ← box wears the nickname "card"
CSS:   .card { background: #14171D; }               ← paint rule for anything named "card"
GLUE:  <link rel="stylesheet" href="style.css">    ← goes in <head>; loads the rules

That is the whole handshake. HTML makes boxes and nicknames them; CSS paints anything with a nickname; one <link> line loads the paint. Every step below is just doing this a lot.

You do not memorise any of this. You copy a working block (this doc is full of them), swap the words, and look at the result in the browser. Every programmer alive keeps a tab open to copy from. Do that and you already are one.

Make it look like The Division

The whole identity is: dark screen, one orange, one cyan, mono labels, corner brackets. Start style.css with colors as variables so every page can share them (and a light theme becomes a 10-line override later):

:root{
  --bg:       #050608;   /* near-black background       */
  --bg-2:     #14171D;   /* card background             */
  --bg-hover: #1C212A;   /* a card when you hover it    */
  --line:     #434B57;   /* borders                     */
  --grid:     #131519;   /* the faint grid lines        */
  --ink:      #E7E9EC;   /* text                        */
  --ink-soft: #9198A1;   /* dimmer text                 */
  --ink-dim:  #5C636B;   /* dimmest text (empty slots)  */
  --accent:     #E8752E; /* THE orange. Use rarely.     */
  --accent-dim: #8C4A1E; /* a darker orange for borders */
  --cyan:     #4FD1D9;   /* secondary. Even rarer.      */
}
body{
  margin:0; background:var(--bg); color:var(--ink);
  font-family:'Public Sans', sans-serif; line-height:1.6;
  /* the faint grid = two 1px gradient lines repeating every 42px */
  background-image:
    linear-gradient(var(--grid) 1px, transparent 1px),
    linear-gradient(90deg, var(--grid) 1px, transparent 1px);
  background-size: 42px 42px;
}
.wrap{ max-width:920px; margin:0 auto; padding:3.5rem 1.5rem; }

The light theme (the "10-line override" this section promised)

Because every colour is a variable, a second theme is just re-declaring the same names for when the page has data-theme="light" on its <html> tag. theme.js (Section J) flips that attribute; this is the palette it switches to. Paste it right under the first :root:

:root[data-theme="light"]{
  --bg:       #F0EEE8;   /* warm paper, not white */
  --bg-2:     #FFFFFF;   /* white cards           */
  --bg-hover: #FFF6ED;   /* faint warm hover      */
  --line:     #ABA48E;   /* muted borders         */
  --grid:     #E0DBCF;   /* faint grid, light ver */
  --ink:      #17181A;   /* near-black text       */
  --ink-soft: #4D4F53;
  --ink-dim:  #83807A;
  --accent:     #C85A1E; /* a hair deeper so orange still reads on white */
  --accent-dim: #E8752E;
  --cyan:     #0D7A82;
}
Why this was missing: the first draft said "a light theme becomes a 10-line override later" and never showed the ten lines. A beginner build-test got stuck inventing light colours — and three real traps surfaced: (1) the grid colour must be a --grid variable (added above), or light mode keeps dark grid lines; (2) a naive colour-invert leaves the orange glaring on white, so light --accent is set a shade deeper on purpose; (3) a later build-test on the easter egg (Section K) reached for --bg-hover, --ink-dim, and --accent-dim and found them undefined, so its buttons and slots rendered with broken colours — all three are now declared here (in both themes) so every later section can lean on them.

G

Pictures & videos

Pictures: shrink before you ship

  • Resize to 1280px wide max, save as JPEG quality ~82. A 2 MB screenshot becomes ~80 KB and nobody can tell.
  • Put them in assets/, use <img src="assets/shot.jpg" loading="lazy" alt="what it shows">.
  • Pick shots that show your work: a character being chased by YOUR ai beats an empty corridor. Diagrams count — a state-machine graph next to the bullet about state machines is proof.

Videos: don't host them, embed YouTube

<div class="yt-embed">  <!-- css: position:relative; padding-top:56.25%; -->
  <iframe src="https://www.youtube.com/embed/VIDEO_ID"
          loading="lazy" allowfullscreen
          style="position:absolute;top:0;left:0;width:100%;height:100%;border:0;"></iframe>
</div>

VIDEO_ID = the 11 characters after watch?v= in the YouTube URL. The padding-top:56.25% box keeps it 16:9 at any width.

Mixed-size picture grids: use columns, not grid

.media-grid{ column-count:2; column-gap:1.1rem; }
.media-grid figure{ break-inside:avoid; margin:0 0 1.1rem; }
@media (max-width:560px){ .media-grid{ column-count:1; } }

Why: CSS grid forces rows to equal height, so a tall portrait next to a short landscape leaves an ugly void. Columns pack them like a masonry wall.

Look at it, then ship it

Test

  • Simplest: right-click the html in Rider → Open in Browser. It opens as a file:// page and needs nothing installed. Optional: if you already have Node or Python, a local server behaves a touch more like the real web — npx serve . (Node) or python -m http.server (Python). Neither is required to build this.
  • Press F12 → toggle the phone icon → check 375px width: nothing should scroll sideways, cards should stack.
  • Click your theme toggle both ways. Hover things in BOTH themes (a hover color that works on dark can read backwards on light).
Contradiction fixed: an earlier draft said "no Node" in Section A but then told you to run npx serve (which needs Node) right here. You need nothing but a browser — the local server is optional.

Ship

  • GitHub Desktop → it lists your changed files → write a summary → Commit to main → Push origin.
  • Wait ~1 minute. Open https://YOURUSERNAME.github.io/. Done. You have a live portfolio.
  • Live site looks broken but files are right? It's the browser hoarding old CSS — hard refresh (Ctrl+F5) and go bump that ?v= number.
I

The rules that keep it good

  • Tell the truth. Write what you actually did, in your own voice. A short honest intro ("this was hard, I learned X") reads better than invented buzzwords — and interviewers ask about everything you claim.
  • Two layers per project: a human intro paragraph (story) + terse factual bullets (scannable). Don't mix them.
  • Orange is loud. The style works because 95% of the page is grayscale. If everything is orange, nothing is.
  • One CSS file. Every page links the same style.css. Fix once, fixed everywhere.
  • Change CSS → bump ?v=. Change a date → change it in both places. Add media → shrink it first. That's the whole maintenance manual.

theme.js — the light/dark switch (whole file)

Two parts. First, the button — drop it wherever you want the toggle (the hero nav is a good spot):

<button class="theme-toggle" onclick="toggleTheme()">theme</button>

Second, the file theme.js itself — pure JavaScript, no HTML tags — loaded once per page with <script src="theme.js"></script> in the <head>:

// theme.js — remember + toggle light/dark

// 1) on load, re-apply whatever was chosen last time
if (localStorage.getItem('theme') === 'light') {
  document.documentElement.setAttribute('data-theme', 'light');
}

// 2) the button calls this
function toggleTheme(){
  var root = document.documentElement;
  if (root.getAttribute('data-theme') === 'light') {
    root.removeAttribute('data-theme');          // back to dark
    localStorage.setItem('theme', 'dark');
  } else {
    root.setAttribute('data-theme', 'light');    // go light
    localStorage.setItem('theme', 'light');
  }
}
Why this section exists: a build-test found the tutorial named theme.js, .section-head, .eyebrow, .hud-list, and the nav .stamp buttons in its examples but never gave their code — so the tester had to invent them, and the invented toggle didn't remember the choice on reload. Everything the steps reference now lives here.

Extra: a hidden Konami-code easter egg

Optional and fun. Clicking the name 19 times quietly opens a little access terminal — a retro on-screen game controller. Punch in the classic Konami code on it (↑ ↑ ↓ ↓ ← → ← → B A) and it reveals links to these very docs. Get it wrong and it shakes and resets. It works for everyone: you can tap the on-screen buttons (phone or desktop) or use your keyboard's arrow keys and B/A. No swiping, no motion sensors, nothing that pops a permission box. Four small pieces.

1. Mark the trigger and build the hidden terminal (HTML)

Take the <h1>My Name</h1> you already made back in Section C and add id="name-egg" to it — that's the only change to your existing heading, and it's the handle JavaScript grabs to count clicks. Then paste the hidden terminal block right after that heading. The hidden attribute keeps it invisible until the 19th click removes it. 2. The rulebook: what the code is, and what each key looks like (add to theme.js)

Two lists. CODE is the secret in order. GLYPH is a lookup that turns a key's name into the symbol to show in a slot. The arrow keys already report themselves as "ArrowUp" etc. when pressed on a keyboard, so we use those exact spellings everywhere — buttons and keyboard speak the same language:

var CODE  = ['ArrowUp','ArrowUp','ArrowDown','ArrowDown',
             'ArrowLeft','ArrowRight','ArrowLeft','ArrowRight','b','a'];
var GLYPH = { ArrowUp:'↑', ArrowDown:'↓', ArrowLeft:'←', ArrowRight:'→', b:'B', a:'A' };

3. Wire it all up (the rest of theme.js)

This is the whole brain. It waits for the page to load, counts the 19 clicks to open the terminal, then feeds every press — from a button or the keyboard — through one press() checker. Right key: fill the next slot and advance. Wrong key: shake and start over. All ten right: reveal the links.

// theme.js loads in the <head>, BEFORE the <h1> exists — so wait for the
// page to finish loading, THEN find everything and wire it up:
document.addEventListener('DOMContentLoaded', function(){
  var nameEgg = document.getElementById('name-egg');
  var term    = document.getElementById('egg-term');
  if (!nameEgg || !term) { return; }   // page has no egg — do nothing

  var CODE  = ['ArrowUp','ArrowUp','ArrowDown','ArrowDown',
               'ArrowLeft','ArrowRight','ArrowLeft','ArrowRight','b','a'];
  var GLYPH = { ArrowUp:'↑', ArrowDown:'↓', ArrowLeft:'←', ArrowRight:'→', b:'B', a:'A' };

  var slots    = term.querySelectorAll('.egg-slot');
  var links    = document.getElementById('egg-links');
  var progress = 0;        // how many correct keys so far
  var unlocked = false;    // stop everything once solved

  // (a) 19 clicks on the name opens the terminal
  var clicks = 0;
  nameEgg.addEventListener('click', function(){
    if (unlocked) { return; }
    clicks++;
    if (clicks === 19) { term.hidden = false; }
  });

  // (b) ONE checker for a single press, no matter where it came from
  function press(key){
    if (unlocked || term.hidden) { return; }
    if (key === CODE[progress]) {                 // right key, next in line
      slots[progress].textContent = GLYPH[key];
      slots[progress].classList.add('on');
      progress++;
      if (progress === CODE.length) { unlock(); } // all ten correct
    } else {
      reset(key);                                 // wrong key
    }
  }

  function reset(key){
    progress = 0;
    for (var i = 0; i < slots.length; i++){
      slots[i].textContent = '';
      slots[i].classList.remove('on');
    }
    // niceity: if the wrong key was itself a valid START (an Up), count it as press 1
    if (key === CODE[0]) {
      slots[0].textContent = GLYPH[key];
      slots[0].classList.add('on');
      progress = 1;
    }
    // restart the shake animation (removing + forcing a reflow + re-adding)
    term.classList.remove('shake');
    void term.offsetWidth;
    term.classList.add('shake');
  }

  function unlock(){
    unlocked = true;
    links.hidden = false;
  }

  // (c) the on-screen gamepad buttons
  term.querySelectorAll('.egg-btn').forEach(function(btn){
    btn.addEventListener('click', function(){
      press(btn.getAttribute('data-key'));
    });
  });

  // (d) desktop keyboard — the bonus path (same press() checker)
  document.addEventListener('keydown', function(e){
    if (term.hidden || unlocked) { return; }
    var k = e.key;
    if (k === 'ArrowUp' || k === 'ArrowDown' || k === 'ArrowLeft' || k === 'ArrowRight') {
      e.preventDefault();     // stop the arrows from scrolling the page
      press(k);
    } else if (k === 'b' || k === 'B') { press('b'); }
    else   if (k === 'a' || k === 'A') { press('a'); }
  });
});
Why the DOMContentLoaded wrapper matters: a build-test once copied egg code to the top of theme.js and it never fired. The reason: theme.js is loaded in the <head>, which runs before the <h1> exists — so getElementById returned null and no listener was ever attached. Wrapping everything in DOMContentLoaded makes it wait until the page (and the name) actually exist.
The keyboard and the buttons both call the exact same press() function — that's the trick that keeps them in perfect sync without writing the logic twice. e.preventDefault() on the arrows stops the page from scrolling while you enter the code.

The one line that isn't optional: .egg-links[hidden]{ display:none; }. A visual build-test found the "Access granted" links showing the instant the terminal opened — before any code was entered. The reason is a classic CSS trap: the browser hides a tag with the hidden attribute using its own rule [hidden]{ display:none }, but that rule is weak, and the moment you write your own .egg-links{ display:flex } your rule wins and the element becomes visible again — attribute or not. Re-stating display:none for the [hidden] case puts the hiding back. (Sneaky detail: in JavaScript links.hidden still reads true, so a code-only test passes while your eyes tell you it's broken — always look at the page.)
Three things make it feel "hidden" yet friendly: the name has no hover cue (cursor:default, no underline) so nobody stumbles in by accident; user-select:none stops the name highlighting into a blue blob while you mash it; and the gamepad is plainly a gamepad, so anyone who knows the code just knows what to do. The click counter and progress are plain variables — refresh the page and both reset to zero, which is perfectly fine for an easter egg.
