How To Build This Portfolio

You. Make website. No framework. Just three files and some pictures. This is the whole path from empty folder to a live site that looks like a video-game HUD, in order, with nothing assumed. Never seen code in your life? The ★ section teaches you to read HTML and CSS in five minutes — it sits right before you type anything.

A

Get tools

  • JetBrains Rider (or any editor) — you write text files in it. That's all a website is.
  • GitHub Desktop + a github.com account — this is how the site gets on the internet, for free.
  • A browser — this is how you look at what you made.
No Node, no React, no build tools. A website is text files. The browser reads them directly. Everything else is optional complexity we are not buying.
B

Make the magic repo

  • On github.com: create a new repository (a project folder GitHub stores online for you — "repo" for short) named exactly YOURUSERNAME.github.io.
  • That exact name is the trick: GitHub automatically serves whatever is in it at https://YOURUSERNAME.github.io/. This is called GitHub Pages. No server, no hosting bill.
  • In GitHub Desktop: Clone repository → pick it → now there's a folder on your computer connected to the internet one.
  • Open that folder in Rider: File → Open → select the folder. Rider treats a plain folder as a project just fine.
★

Before you type: read the two languages

The next steps ask you to type code. First learn to read it. A website is only two languages, and neither is scary. Five minutes here and every code block below stops looking like magic runes.

The whole idea in one breath

HTML builds bare boxes and puts your words in them. CSS is a list of paint rules for those boxes. One line links them. That is the entire craft — everything else is repetition.

HTML = the boxes (structure)

A web page is boxes stacked and nested inside boxes. HTML draws them with tags. A tag looks like this:

<h1>My Name</h1>
  • <h1> — the opening tag: "a big heading starts here."
  • My Name — the content a human actually sees.
  • </h1> — the closing tag. The slash / means "the heading ends here."

Four rules cover almost all of it:

  • Most tags come in a pair (open + close). Whatever sits between them is what shows up.
  • A few tags are loners — they hold nothing and never close: <img> (a picture), <link>, <meta>.
  • Boxes go inside boxes: <div><p>hi</p></div> is a box holding a paragraph.
  • Tags can carry extra notes called attributes, written inside the <>:
<a href="about.html">About</a>      <!-- href = WHERE the link goes -->
<img src="cat.jpg" alt="a cat">     <!-- src = WHICH picture, alt = describe it -->
<div class="card"> ... </div>        <!-- class = a NICKNAME (the important one) -->
class="card" is a nickname you stick on a box so the paint file can find it later. This one attribute is how HTML and CSS hold hands — remember it above all else.

Every page has the same skeleton: a head (invisible setup) and a body (everything you see):

<head>   ← page title, font links, the link to your style file. Nobody SEES this.
<body>   ← headings, text, pictures, cards. Everything a human looks at.

The dozen tags you will actually use:

  • <h1> <h2> <h3> — headings, biggest to smaller.
  • <p> — a paragraph of text.
  • <div> — a plain box. The workhorse; it holds other things.
  • <span> — a small inline piece (to colour one word inside a sentence).
  • <a> — a link · <img> — a picture.
  • <ul> + <li> — a bullet list and each bullet in it.
  • <button> — a clickable button.

CSS = the paint (looks)

HTML gave you bare boxes. CSS is a list of rules, each one saying "find these boxes, make them look like this." One rule:

.card {
  background: #14171D;
  padding: 1rem;
}
  • .card — the selector: WHO to paint.
  • { } — the paint goes inside the curly braces.
  • background: #14171D; — one instruction: property : value ;. Every line ends in a semicolon.

How to point at boxes (selectors):

  • h1 { } — every h1 heading on the page.
  • .card { } — every box wearing class="card". The dot means "nickname."
  • body { } — the whole page.

The paint instructions (properties) you will use most:

  • color — text colour · background — fill behind the box.
  • padding — cushion inside · margin — gap outside.
  • border — the outline line · font-family — which font.
  • width / max-width — how wide · display: flex — lay the children in a row.

Knowing the box model is most of not fighting CSS. Every box is four nested layers, from the inside out:

MARGIN — gap OUTSIDE (space to neighbours)
BORDER — the visible outline
PADDING — cushion INSIDE
CONTENT
your text / picture

Colours like #E8752E are just a colour's secret code (called "hex"). You never memorise them — you copy them.

Variables are the one clever trick, and they save your life. Name a colour once at the very top:

:root {
  --accent: #E8752E;   /* name the orange once */
}

Then anywhere you want it, write var(--accent) instead of the code. Change the colour in that one place and it updates on every page at once. That is why the whole site shares a single look.

How the two hold hands

HTML:  <div class="card">...</div>                 ← box wears the nickname "card"
CSS:   .card { background: #14171D; }               ← paint rule for anything named "card"
GLUE:  <link rel="stylesheet" href="style.css">    ← goes in <head>; loads the rules

That is the whole handshake. HTML makes boxes and nicknames them; CSS paints anything with a nickname; one <link> line loads the paint. Every step below is just doing this a lot.

You do not memorise any of this. You copy a working block (this doc is full of them), swap the words, and look at the result in the browser. Every programmer alive keeps a tab open to copy from. Do that and you already are one.

Why this grew: the first draft showed only an <h1> and one <p> here, then later sections assumed a hero, nav, and sections already existed — a beginner build-test had nowhere to hang the cards. This skeleton is the missing frame. Two more build-tests then noticed the <head> never loaded the fonts Section D relies on (so their pages fell back to plain system text), so the font <link> now lives in the skeleton too.
The new head/body lines aren't decoration: <meta name="description"> is the sentence search engines show; the two preconnect lines make fonts load faster (Section D); and skip-link + <header> + <main id="main"> + <footer> are the accessibility bones (Section L explains each in one line). The theme button is icon-only, so it carries aria-label and no onclick — the switch is wired up in Section J.
See the ?v=1 after both style.css and theme.js? Remember it. Every time you change that file, bump it: ?v=2, ?v=3... Browsers hoard old copies like a caveman hoards rocks; changing the number forces a fresh fetch. The two counts are independent — change the CSS, bump the CSS one; change the script's behaviour, bump the theme.js one. Skip this and one day your live site looks broken while your files are fine.
D

Make it look like The Division

The whole identity is: dark screen, one orange, one cyan, mono labels, corner brackets. Start style.css with colors as variables so every page can share them (and a light theme becomes a 10-line override later):

:root{
  --bg:#050608; --bg-2:#14171D; --bg-3:#1C212A;       /* three dark surfaces           */
  --line:#272C35; --line-2:#434B57; --grid:#131519;   /* borders + faint grid          */
  --hatch:rgba(255,255,255,0.035); --shadow:rgba(0,0,0,0.45);
  --ink:#E7E9EC; --ink-soft:#9198A1; --ink-dim:#7E858F;   /* text, dimmer, dimmest     */
  --accent:#E8752E; --accent-dim:#8C4A1E; --accent-glow:rgba(232,117,46,0.22); /* THE orange (rare) */
  --cyan:#4FD1D9; --bg-hover:#1C212A; --max:920px;    /* secondary (rarer)             */
  /* fonts named once, reused as var(--mono) etc. */
  --mono:'IBM Plex Mono', ui-monospace, Consolas, monospace;
  --display:'Rajdhani', 'Arial Narrow', sans-serif;
  --sans:'Public Sans', system-ui, -apple-system, sans-serif;
}
body{
  margin:0; background:var(--bg); color:var(--ink);
  font-family:'Public Sans', sans-serif; line-height:1.6;
  /* the faint grid = two 1px gradient lines repeating every 42px */
  background-image:
    linear-gradient(var(--grid) 1px, transparent 1px),
    linear-gradient(90deg, var(--grid) 1px, transparent 1px);
  background-size: 42px 42px;
}
.wrap{ max-width:920px; margin:0 auto; padding:3.5rem 1.5rem; }

The light theme (the "10-line override" this section promised)

Because every colour is a variable, a second theme is just re-declaring the same names for when the page has data-theme="light" on its <html> tag. theme.js (Section J) flips that attribute; this is the palette it switches to. Paste it right under the first :root:

:root[data-theme="light"]{
  --bg:#F0EEE8; --bg-2:#FFFFFF; --bg-3:#E7E3D9;        /* warm paper, white cards       */
  --line:#D0CABA; --line-2:#ABA48E; --grid:#E0DBCF;
  --hatch:rgba(0,0,0,0.045); --shadow:rgba(46,40,28,0.14);
  --ink:#17181A; --ink-soft:#4D4F53; --ink-dim:#6A6760;
  --accent:#C85A1E; --accent-dim:#E8752E; --accent-glow:rgba(200,90,30,0.14); /* deeper so orange reads on white */
  --cyan:#0D7A82; --bg-hover:#FFF6ED;
}
Besides the colours, that block also names a few structural tokens the later sections lean on — --bg-3 (a third surface), --line-2 (a brighter border), --accent-glow (the soft halo), --hatch/--shadow, and the three font vars (--mono, --display, --sans). Paste the whole block — Sections J and M and the easter egg all use them.
Why this was missing: the first draft said "a light theme becomes a 10-line override later" and never showed the ten lines. A beginner build-test got stuck inventing light colours — and three real traps surfaced: (1) the grid colour must be a --grid variable (added above), or light mode keeps dark grid lines; (2) a naive colour-invert leaves the orange glaring on white, so light --accent is set a shade deeper on purpose; (3) a later build-test on the easter egg (Section K) reached for --bg-hover, --ink-dim, and --accent-dim and found them undefined, so its buttons and slots rendered with broken colours — all three are now declared here (in both themes) so every later section can lean on them.

Fonts (paste in <head> of every page)

Three lines, in this order:

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Rajdhani:wght@600;700&family=Public+Sans:wght@400;500&family=IBM+Plex+Mono:wght@400;500&display=swap" rel="stylesheet">
  • Rajdhani = big shouty uppercase titles.
  • IBM Plex Mono = anything that should feel like machine data: labels, dates, chips.
  • Public Sans = normal reading text.
  • The two preconnect lines are a head-start handshake: they tell the browser "you'll need Google's font server in a second, start the connection now" so fonts arrive faster. (crossorigin on the second is required because the font files come from a different sub-server.)
  • display=swap at the end of the URL = "show my text in a backup font immediately, then swap to the real font once it loads" — so words are never invisible while fonts download.

The signature move: corner brackets

A box with tiny orange L-shapes on two opposite corners. It's just a border plus two positioned pseudo-elements (::before / ::after = extra shapes CSS draws on a box without adding any HTML; here they're the two corner Ls):

.panel{ position:relative; border:1px solid var(--line); background:var(--bg-2); padding:1.6rem; }
.panel::before, .panel::after{ content:""; position:absolute; width:10px; height:10px; }
.panel::before{ top:-1px; left:-1px;   border-top:2px solid var(--accent); border-left:2px solid var(--accent); }
.panel::after { bottom:-1px; right:-1px; border-bottom:2px solid var(--accent); border-right:2px solid var(--accent); }

Foundation polish (two tiny extras on body)

/* soft orange glow bleeding down from the top */
body::before{ content:""; position:fixed; inset:0; pointer-events:none; z-index:0;
  background:radial-gradient(120% 90% at 50% -10%, var(--accent-glow), transparent 60%); opacity:0.5; }
:root[data-theme="light"] body::before{ opacity:0.3; }

/* highlight colour when you drag-select text */
::selection{ background:var(--accent); color:var(--bg); }

The first paints the faint orange wash at the top of every page; the second makes selected text orange-on-dark instead of the browser's default blue.

COMPANY NAMESHIPPED2022–23

PICTURE 2 (optional)

Project Title

>> WHAT I WORKED ON
> thing one

The layout trick is one line of flexbox (CSS's way of lining boxes up in a row or column, from the ★ section) — media gets a slightly bigger share:

.entry-body { display:flex; gap:1.2rem; padding:1rem 1.2rem; }
.entry-media{ flex:1.15; display:flex; flex-direction:column; gap:0.9rem; }  /* 1.15 > 1 = a tad bigger */
.entry-text { flex:1; }
.entry-media img{ width:100%; max-height:250px; object-fit:cover; border:1px solid var(--line); }

/* phone: stack instead of squeeze */
@media (max-width:560px){ .entry-body{ flex-direction:column; } }
Two little CSS words in that block: object-fit:cover = crop the image to fill its box without squishing it (its cousin contain fits the whole image inside instead, which can leave gaps). And @media (max-width:560px) is a "phone rule" — below 560px the card stacks top-to-bottom. You'll meet a second breakpoint, 620px, on other components later; that's on purpose — each piece stacks at whatever width it starts to feel cramped. Copy the numbers as given.

G

Pictures & videos

Pictures: shrink before you ship

  • Resize to 1280px wide max, save as JPEG quality ~82. A 2 MB screenshot becomes ~80 KB and nobody can tell.
  • Put them in assets/, use <img src="assets/shot.jpg" loading="lazy" alt="what it shows">.
  • Pick shots that show your work: a character being chased by YOUR ai beats an empty corridor. Diagrams count — a state-machine graph next to the bullet about state machines is proof.
No real screenshots yet? Drop any placeholder into assets/ (even a plain solid-colour PNG or a grey box) so the layout isn't broken while you build — swap the real shots in later. A broken-image icon makes the whole page look broken, so never ship one.

Videos: don't host them, embed YouTube

<div class="yt-embed">  <!-- css: position:relative; padding-top:56.25%; -->
  <iframe src="https://www.youtube.com/embed/VIDEO_ID"
          loading="lazy" allowfullscreen
          style="position:absolute;top:0;left:0;width:100%;height:100%;border:0;"></iframe>
</div>

VIDEO_ID = the 11 characters after watch?v= in the YouTube URL. The padding-top:56.25% box keeps it 16:9 (the standard widescreen shape — 56.25% is just 9 ÷ 16) at any width.

Mixed-size picture grids: use columns, not grid

Wrap the shots in a .media-grid, one <figure> per picture:

<div class="media-grid">
  <figure><img src="assets/a.jpg" loading="lazy" alt="what it shows"><figcaption>caption</figcaption></figure>
  <figure><img src="assets/b.jpg" loading="lazy" alt="what it shows"></figure>
</div>
.media-grid{ column-count:2; column-gap:1.1rem; }
.media-grid figure{ break-inside:avoid; margin:0 0 1.1rem; }
@media (max-width:560px){ .media-grid{ column-count:1; } }

Why: CSS grid forces rows to equal height, so a tall portrait next to a short landscape leaves an ugly void. Columns pack them like a masonry wall.

H

Look at it, then ship it

Test

  • Simplest: right-click the html in Rider → Open in Browser. It opens as a file:// page and needs nothing installed. Optional: if you already have Node or Python, a local server behaves a touch more like the real web — npx serve . (Node) or python -m http.server (Python). Neither is required to build this.
  • Press F12 → toggle the phone icon → check 375px width: nothing should scroll sideways, cards should stack.
  • Click your theme toggle both ways. Hover things in BOTH themes (a hover color that works on dark can read backwards on light).
Contradiction fixed: an earlier draft said "no Node" in Section A but then told you to run npx serve (which needs Node) right here. You need nothing but a browser — the local server is optional.

Ship

  • GitHub Desktop → it lists your changed files → write a summary → Commit to main (save a snapshot of the changes) → Push origin (upload that snapshot to GitHub, which makes it live).
  • Wait ~1 minute. Open https://YOURUSERNAME.github.io/. Done. You have a live portfolio.
  • Live site looks broken but files are right? It's the browser hoarding old CSS — hard refresh (Ctrl+F5) and go bump that ?v= number.
Optional — a favicon (the tiny icon in the browser tab): drop a favicon.ico in the repo root, or add <link rel="icon" href="favicon.png"> to each page's <head>. This site doesn't ship one; add it only if you want the tab to show your mark.
I

The rules that keep it good

  • Tell the truth. Write what you actually did, in your own voice. A short honest intro ("this was hard, I learned X") reads better than invented buzzwords — and interviewers ask about everything you claim.
  • Two layers per project: a human intro paragraph (story) + terse factual bullets (scannable). Don't mix them.
  • Orange is loud. The style works because 95% of the page is grayscale. If everything is orange, nothing is.
  • One CSS file. Every page links the same style.css. Fix once, fixed everywhere.
  • Change CSS → bump ?v=. Change a date → change it in both places. Add media → shrink it first. That's the whole maintenance manual.
J

Copy-paste appendix (the shared bits every page leans on)

The steps above focused on the interesting parts. These are the small shared pieces they keep referencing — paste them into style.css once and every page gets them.

Headings, eyebrow, section headers, bullets, nav stamps

One thing to know before pasting: clamp(min, ideal, max) on the h1 = a font size that grows and shrinks with the screen width but never below min or past max (so the title fits phones and desktops without extra rules).

h1{ font-family:'Rajdhani', sans-serif; font-weight:700; font-size:clamp(2.2rem,6vw,3.2rem);

theme.js — the shared script (toggle + copy-email + back-to-top)

theme.js loads once per page (<script src="theme.js?v=1"></script> in the <head>) and does three jobs; the Konami egg (Section K) splices a fourth into the same file. The button itself carries no text and no onclick — just an accessible label:

<button class="theme-toggle" type="button" aria-label="Toggle light/dark theme"></button>

Part 1 — remember the theme (runs right away). (localStorage = the browser's tiny memory box that survives a page refresh; we stash the chosen theme there.)

// Re-apply whatever was chosen last time. DARK is the default: we only set a
// theme when one was saved, and we never follow the OS setting (see Section L).
(function(){
  var saved = localStorage.getItem('theme');
  if (saved === 'light' || saved === 'dark') {
    document.documentElement.setAttribute('data-theme', saved);
  }
})();

Part 2 — the rest waits for the page, then wires the buttons up:

document.addEventListener('DOMContentLoaded', function(){

  // (a) light/dark toggle — every .theme-toggle button flips the theme + remembers it
  document.querySelectorAll('.theme-toggle').forEach(function(btn){
    btn.addEventListener('click', function(){
      var current = document.documentElement.getAttribute('data-theme') === 'light' ? 'light' : 'dark';
      var next = current === 'light' ? 'dark' : 'light';
      document.documentElement.setAttribute('data-theme', next);
      localStorage.setItem('theme', next);
    });
  });

  // (b) copy-email — click a [data-copy] element, its address jumps to the clipboard
  document.querySelectorAll('[data-copy]').forEach(function(el){
    var tip = el.querySelector('.copy-tooltip');
    if (!tip) return;
    var original = tip.textContent;
    el.addEventListener('click', function(){
      if (!navigator.clipboard) { tip.textContent = el.getAttribute('data-copy'); return; }  // old browser: just show it
      navigator.clipboard.writeText(el.getAttribute('data-copy')).then(function(){
        tip.textContent = 'Copied!'; tip.classList.add('copied');
      }).catch(function(){
        tip.textContent = 'Press Ctrl+C to copy';   // blocked — tell them how instead
      });
    });
    el.addEventListener('mouseleave', function(){
      setTimeout(function(){ tip.textContent = original; tip.classList.remove('copied'); }, 200);
    });
  });

  // (c) back-to-top — build the button, reveal it after 600px, smooth-scroll home
  var toTop = document.createElement('button');
  toTop.className = 'to-top'; toTop.type = 'button';
  toTop.setAttribute('aria-label', 'Back to top');
  toTop.textContent = '▲ top';
  document.body.appendChild(toTop);
  toTop.addEventListener('click', function(){ window.scrollTo({ top:0, behavior:'smooth' }); });
  var onScroll = function(){ toTop.classList.toggle('show', window.scrollY > 600); };
  window.addEventListener('scroll', onScroll, { passive:true });
  onScroll();

});
Why addEventListener, not onclick="…": the buttons carry no onclick in the HTML; the script attaches the behaviour here. That's the tidy standard ("unobtrusive JS") — the markup stays clean and one script wires every page. It's also why the toggle button holds no text: the CSS ::before draws the sun/moon label.
Copy-email asks for nothing: navigator.clipboard.writeText copies with no popup on a normal click. The two guards — the if (!navigator.clipboard) fallback and the .catch(...) — cover ancient browsers and the rare blocked case by just showing the address to copy by hand. No permission box, ever (a hard rule for this site).
Why this section exists: a build-test found the tutorial naming theme.js, .section-head, .eyebrow, .hud-list, .stamp, .copy-tooltip, and .to-top in its examples but never giving their code — so the tester invented them, and the invented toggle didn't remember the choice on reload. Everything the steps reference now lives here.

Extra: a hidden Konami-code easter egg

Optional and fun. Clicking the name 19 times quietly opens a little access terminal — a retro on-screen game controller. Punch in the classic Konami code on it (↑ ↑ ↓ ↓ ← → ← → B A) and it reveals links to these very docs. Get it wrong and it shakes and resets. 2. The rulebook: what the code is, and what each key looks like (add to theme.js)

Two lists. CODE is the secret in order. GLYPH is a lookup that turns a key's name into the symbol to show in a slot. The arrow keys already report themselves as "ArrowUp" etc. when pressed on a keyboard, so we use those exact spellings everywhere — buttons and keyboard speak the same language:

var CODE  = ['ArrowUp','ArrowUp','ArrowDown','ArrowDown',
             'ArrowLeft','ArrowRight','ArrowLeft','ArrowRight','b','a'];
var GLYPH = { ArrowUp:'↑', ArrowDown:'↓', ArrowLeft:'←', ArrowRight:'→', b:'B', a:'A' };

3. Wire it all up (the rest of theme.js)

This is the whole brain. It waits for the page to load, counts the 19 clicks to open the terminal, then feeds every press — from a button or the keyboard — through one press() checker. Right key: fill the next slot and advance. Wrong key: shake and start over. All ten right: reveal the links.

// theme.js loads in the <head>, BEFORE the <h1> exists — so wait for the
// page to finish loading, THEN find everything and wire it up:
document.addEventListener('DOMContentLoaded', function(){
  var nameEgg = document.getElementById('name-egg');
  var term    = document.getElementById('egg-term');
  if (!nameEgg || !term) { return; }   // page has no egg — do nothing

  var CODE  = ['ArrowUp','ArrowUp','ArrowDown','ArrowDown',
               'ArrowLeft','ArrowRight','ArrowLeft','ArrowRight','b','a'];
  var GLYPH = { ArrowUp:'↑', ArrowDown:'↓', ArrowLeft:'←', ArrowRight:'→', b:'B', a:'A' };

  var slots    = term.querySelectorAll('.egg-slot');
  var links    = document.getElementById('egg-links');
  var progress = 0;        // how many correct keys so far
  var unlocked = false;    // stop everything once solved

  // (a) 19 clicks on the name opens the terminal
  var clicks = 0;
  nameEgg.addEventListener('click', function(){
    if (unlocked) { return; }
    clicks++;
    if (clicks === 19) { term.hidden = false; }
  });

  // (b) ONE checker for a single press, no matter where it came from
  function press(key){
    if (unlocked || term.hidden) { return; }
    if (key === CODE[progress]) {                 // right key, next in line
      slots[progress].textContent = GLYPH[key];
      slots[progress].classList.add('on');
      progress++;
      if (progress === CODE.length) { unlock(); } // all ten correct
    } else {
      reset(key);                                 // wrong key
    }
  }

  function reset(key){
    progress = 0;
    for (var i = 0; i < slots.length; i++){
      slots[i].textContent = '';
      slots[i].classList.remove('on');
    }
    // niceity: if the wrong key was itself a valid START (an Up), count it as press 1
    if (key === CODE[0]) {
      slots[0].textContent = GLYPH[key];
      slots[0].classList.add('on');
      progress = 1;
    }
    // restart the shake animation (removing + forcing a reflow + re-adding)
    term.classList.remove('shake');
    void term.offsetWidth;
    term.classList.add('shake');
  }

  function unlock(){
    unlocked = true;
    term.classList.add('solved');   // hides the gamepad (see the CSS in step 4)
    links.hidden = false;
  }

  // (c) the on-screen gamepad buttons
  term.querySelectorAll('.egg-btn').forEach(function(btn){
    btn.addEventListener('click', function(){
      press(btn.getAttribute('data-key'));
    });
  });

  // (d) desktop keyboard — the bonus path (same press() checker)
  document.addEventListener('keydown', function(e){
    if (term.hidden || unlocked) { return; }
    var k = e.key;
    if (k === 'ArrowUp' || k === 'ArrowDown' || k === 'ArrowLeft' || k === 'ArrowRight') {
      e.preventDefault();     // stop the arrows from scrolling the page
      press(k);
    } else if (k === 'b' || k === 'B') { press('b'); }
    else   if (k === 'a' || k === 'A') { press('a'); }
  });
});
Why the DOMContentLoaded wrapper matters: a build-test once copied egg code to the top of theme.js and it never fired. The reason: theme.js is loaded in the <head>, which runs before the <h1> exists — so getElementById returned null and no listener was ever attached. Wrapping everything in DOMContentLoaded makes it wait until the page (and the name) actually exist.
The keyboard and the buttons both call the exact same press() function — that's the trick that keeps them in perfect sync without writing the logic twice. e.preventDefault() on the arrows stops the page from scrolling while you enter the code.

The one line that isn't optional: .egg-links[hidden]{ display:none; }. A visual build-test found the "Access granted" links showing the instant the terminal opened — before any code was entered. The reason is a classic CSS trap: the browser hides a tag with the hidden attribute using its own rule [hidden]{ display:none }, but that rule is weak, and the moment you write your own .egg-links{ display:flex } your rule wins and the element becomes visible again — attribute or not. Re-stating display:none for the [hidden] case puts the hiding back. (Sneaky detail: in JavaScript links.hidden still reads true, so a code-only test passes while your eyes tell you it's broken — always look at the page.)
Three things make it feel "hidden" yet friendly: the name has no hover cue (cursor:default, no underline) so nobody stumbles in by accident; user-select:none stops the name highlighting into a blue blob while you mash it; and the gamepad is plainly a gamepad, so anyone who knows the code just knows what to do. The click counter and progress are plain variables — refresh the page and both reset to zero, which is perfectly fine for an easter egg.

L

Accessibility (a11y)

What "a11y" means: a11y = a-ccessibilit-y (an "a", then 11 letters, then "y" — a nerd shortcut for a long word). It means the site also works for people who can't use it the normal way: folks on a screen reader (software that reads the page aloud), keyboard-only users, and low-vision users. None of it changes how the site looks — a handful of extra tags and one CSS block.

Everything below is on the real site. Add it and your build matches; skip it and you fail both the standard and the control.

1 · Landmarks — name the big regions

Screen readers let people jump between landmarks (the page's major zones). You get them almost free by using the right tags instead of a plain <div>:

<body>
  <a class="skip-link" href="#main">Skip to main content</a>
  <div class="wrap">
    <header> … masthead / hero … </header>
    <main id="main">
      <section> … </section>
    </main>
    <footer> … </footer>
  </div>
</body>
  • <header> = the top block · <main id="main"> = THE one main area (exactly one per page) · <section> = a chunk of it · <nav> = a set of links · <footer> = the bottom block.
  • Give a <nav> a name when there's more than one: <nav class="topbar" aria-label="Breadcrumb">.

2 · Skip link

<a class="skip-link" href="#main">Skip to main content</a>
.skip-link{ position:absolute; left:0.5rem; top:-3rem; z-index:100;   /* parked off-screen */
  font-family:var(--mono); font-size:0.8rem; background:var(--accent); color:var(--bg);
  padding:0.5rem 0.9rem; text-decoration:none; transition:top 0.15s ease; }
.skip-link:focus{ top:0.5rem; }   /* slides in only when tabbed to */

Invisible until a keyboard user presses Tab; then it appears so they can skip straight to #main instead of tabbing through every header link.

3 · Icon-only buttons need a text label

<button class="theme-toggle" type="button" aria-label="Toggle light/dark theme"></button>
aria-label = a hidden label read aloud by screen readers. Use it whenever a control shows an icon but no text.

4 · Images: alt text (and when to leave it empty)

<img src="assets/shot.jpg" alt="Guard chasing the player past my AI patrol">   <!-- meaningful -->
<img src="assets/linkedin.png" alt="">   <!-- decorative: text nearby already says "LinkedIn" -->

alt describes the picture for people who can't see it. If it's purely decorative and the words next to it already say the same thing, use an empty alt="" so the reader skips it.

5 · Reduced motion

Some people get motion-sick; their device has a "reduce motion" switch. Honour it with one block at the end of style.css:

@media (prefers-reduced-motion: reduce){
  html{ scroll-behavior:auto; }
  .entry h3 a.glitch:hover{ animation:none; }   /* glitch stays for everyone else */
  .entry-status.st-live::before{ animation:none; }   /* the pulsing dot */
  .egg-term, .egg-term.shake{ animation:none; }
  .icon-row .icon-chip:hover, .software-icons .icon-chip:hover{ transform:none; }
  .stamp:active, .egg-btn:active{ transform:none; }
  .skip-link{ transition:none; }
}
prefers-reduced-motion = a setting the browser reports; the code says "if they asked for less motion, switch the moving parts off." The glitch and pulse still play for everyone who didn't ask.

6 · Keyboard focus ring

a:focus-visible, button:focus-visible{ outline:1px solid var(--accent); outline-offset:3px; border-radius:2px; }

When someone tabs with the keyboard, an orange outline marks the focused link/button. :focus-visible shows it for keyboard users but not on mouse clicks.

7 · Safe external links

<a href="https://example.com" target="_blank" rel="noopener">Site &#8599;</a>

target="_blank" opens a new tab; rel="noopener" goes with it — one word that stops the new page from reaching back and messing with yours. Always pair them.

8 · Two more that live elsewhere

  • Lazy images — loading="lazy" (Section G) makes the page show sooner.
  • Real dates — wrap dates in <time> (Sections E and M); marks them as machine-readable, not just text.

9 · Colour contrast

Dim grey text on a dark card can drop below the readable line, and there's an official rule for it. WCAG (say it "W-cag") is short for Web Content Accessibility Guidelines — the world's checklist for making a website usable by everyone. It's written by the W3C, the international group that makes the web's rulebooks (the same people who decide how HTML itself works). WCAG sorts a site into three grades:

  • A — the bare minimum.
  • AA — the level real sites and most accessibility laws expect. This is the one to aim for.
  • AAA — the strictest, gold-plated level.

So "WCAG AA" just means "meets that middle grade." The main thing AA asks for here: normal text should stand out from its background by about 4.5 to 1 in brightness. The site's dimmest ink (--ink-dim) was nudged brighter for exactly this reason.

Light/dark and the system setting: there's a browser feature (prefers-color-scheme) that follows the visitor's OS light/dark setting. This site deliberately does not use it — it's a dark-brand ("Division HUD") that always opens dark, and the toggle lets anyone switch. Following the system setting here would fight the brand. For a non-branded site, following it is a fine choice.
