03

Design system

Palette (dark theme)

--bg #050608
--bg-2 #14171D (cards)
--bg-3 / --bg-hover
--accent #E8752E
--cyan #4FD1D9
--ink #E7E9EC

Light theme overrides live in :root[data-theme="light"] — warm paper (#F0EEE8) with white cards. Special case: --bg-hover is #1C212A in dark (hover = lift) but #FFF6ED in light (hover = warm glow) — a plain darker hover reads as "dimmed" on white cards and confuses which card is selected.

Fonts

  • Rajdhani (600/700) — big uppercase display headers.
  • IBM Plex Mono — data readouts, labels, chips, dates, everything "HUD".
  • Public Sans — body copy.

Page anatomy

Every project sub-page follows this pattern

topbar (← back · breadcrumb · theme toggle)
header
  ├── .eyebrow        "WORK EXPERIENCE · 2022–23"   ← date lives HERE and on the index card. Update both!
  ├── h1              project title
  ├── .lede           one-line "what the game/app is"
  ├── .panel          ── the briefing panel ──
  │     ├── >> INTRODUCTION  first-person narrative (the Wix voice: learning / ownership / rework story)
  │     └── .specs           Role · Team · Duration · Engine
  └── .stamps         store links / Back
sections 01..N        Gameplay first, then systems, then Media last
  └── ul.hud-list     terse factual bullets (the scannable layer)
footer

The two-layer content rule: the Introduction carries the personal story (first-person, faithful to the original Wix narrative — challenge, luck, learning, rework), while the bullets stay terse and factual. Never mix the two: no manufactured enthusiasm in bullets, no spec-dumps in the intro.

Content principles

  • Info + narrative must both match the source. Facts from the Wix page (fetch the specific sub-page URL, not the SPA landing page); tone from its first-person intros. Owner's direct statements override everything.
  • League of Bomb is capability-level only — no internal class names or architecture. NDA-safe by design; keep it that way.
  • Dates live in two places — the index card and the sub-page eyebrow. They have drifted apart before; always change both.
  • Don't invent. Every "hazard scenarios"-style embellishment found during cross-checks got removed. If a fact is missing, ask the owner (plain-text questions, one at a time — not batched forms).
  • Media is optimized, not hot-linked raw — local assets resized to ≤1280px, JPEG q82–84 (45–150 KB each). Legacy Wix media still hot-links from static.wixstatic.com; migrating those to local optimized copies is a sensible future step.

Maintenance guide

Add a new project

  • Copy the closest existing page in projects/ (e.g. replace.html), replace: title, meta description, breadcrumb, eyebrow date, lede, Introduction, specs, sections, media.
  • Add an index card in the right chronological slot: .entry with company, status badge, year + duration, up to 2 media, title link, >> WHAT I WORKED ON bullets, stack chips.
  • If it's a new employer, add an .xp-list row in About.

Change styling

  • Edit style.css, then bump ?v= on all 11 pages (find-replace style.css?v=OLD → style.css?v=NEW).
  • Check both themes (toggle) and mobile width (<560px behaviors: media stacks, breadcrumb hides, media-grid goes 1-column).

Publish

  • Review in GitHub Desktop → commit → push → GitHub Pages deploys in ~1 min. If the live site looks unstyled: hard refresh (Ctrl+F5) — it's a cached stylesheet, not broken code.
